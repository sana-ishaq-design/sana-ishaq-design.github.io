A barcode generator that works

01 — Project Overview

No account. No server.
No tab switch required.

Barcode Studio lets users generate QR codes and linear barcodes — EAN-13, Code 128, UPC-A — directly inside a Chrome popup and export them as PNG, SVG, WebP, or PDF, individually or in bulk. All processing is local. No data ever leaves the device.

The product serves a precise moment: a user is mid-task, needs a barcode in seconds, and does not want the overhead of a new tab, an account, or an external upload. Every UX decision was evaluated against one question: does this help a user complete the task in under 30 seconds?

Product scope

Formats: QR Code · EAN-13 · Code 128 · UPC-A  ·  Export: PNG · SVG · WebP · PDF  ·  Modes: Single + Batch  ·  Themes: Dark + Light (auto)  ·  Processing: 100% local

02 — The Challenge

The existing ecosystem fails users
in three consistent ways.

A competitive audit of web tools, desktop software, and existing Chrome extensions revealed the same friction patterns repeating across every category.

Challenge 01
⇄
Context Switching
Every existing solution requires leaving the active workflow — opening a new tab, navigating to an external site, and returning with the asset. The need arises in context; the tool forces users out of it.
Challenge 02
⚑
Privacy Concerns
Product codes, internal URLs, and confidential identifiers are routinely uploaded to remote servers with opaque data retention policies. Most users don't realise this is happening at all.
Challenge 03
◈
Overly Complex Tools
Solutions are either bloated desktop software with far more features than needed, or minimal browser tools limited to a single format with no export control and inconsistent quality.
Design challenge

How might we design a barcode generator that works entirely inside the browser, supports common formats, enables task completion in under 30 seconds, and keeps all data strictly local?

03 — Why a Chrome Extension?

Three reasons the format
was the solution.

A web app would require a tab switch. A desktop tool requires an install. A Chrome extension lives exactly where users already are — and that changes everything about the experience.

⊙
Stay In Context
Users generate barcodes without leaving their current workflow. The extension lives in the browser toolbar and opens as a popup — zero navigation overhead, zero tab switching.
◎
Protect Privacy
Running locally inside Chrome means no server request is ever made. Sensitive data — product SKUs, internal URLs, confidential codes — never leaves the device. Privacy is architectural, not a policy.
◐
Move Faster
A focused popup UI with a single task — generate and export — eliminates every distraction present in full web tools. The constraint of the popup format enforces exactly the right level of simplicity.

07 — Key UX Decisions

Six decisions that define
how the extension feels to use.

Extension design operates inside hard constraints — a 400×600px popup with no persistent state. Every decision below was chosen over a documented alternative, with explicit reasoning tied to user research or documented assumptions.

Decision 01
Format Chips vs Dropdown
Challenge
A dropdown hides all options behind one interaction, requiring users to recall available formats before choosing.
Decision
Quick-select chips display all four formats simultaneously. One tap selects; active state always visible.
Reasoning
Reduces format selection from 2 interactions to 1. Supports A2: most users select one format per session.
Decision 02
Unified Export — Single & Batch
Challenge
Separate screens for single and batch export add navigation overhead for a decision that takes seconds.
Decision
One Export tab with progressive disclosure — single export is primary, batch (CSV→ZIP) is immediately accessible but secondary.
Reasoning
Single-tab architecture removes navigation. Primary flow stays clean. Batch capability remains discoverable.
Decision 03
Privacy Messaging at Point of Input
Challenge
Privacy-conscious users (A5) will abandon if they can't verify local processing — but a modal adds friction.
Decision
"100% Local · Never sent to server" placed directly below the payload input field.
Reasoning
Addresses trust concern exactly where it forms — at input — without interrupting the flow with a modal or help screen.
Decision 04
Auto Theme as Default
Challenge
Manual theme choice adds friction on first open. A settings screen adds navigation depth to a popup with no navigation bar.
Decision
Extension auto-detects system theme (dark/light) and applies it without user input. Manual override available in the About overlay.
Reasoning
Eliminates a settings page entirely. Enforces a proper CSS token system — all values must use semantic variables, not hardcoded hex.
Decision 05
About Overlay vs Settings Page
Challenge
A settings page requires navigation, back button, and persistent state — costly inside a popup viewport with no browser chrome.
Decision
ℹ icon opens an in-context overlay — keyboard shortcuts, open source links, developer credit, and feedback access without leaving the active screen.
Reasoning
Maintains single-screen architecture. Overlay dismisses cleanly. No navigation state to maintain or restore.
Decision 06
Popup-Optimised Typography
Challenge
Standard web type scales (16px body, 24–48px headings) fail at 400×600px — everything reads either too large or too small.
Decision
Custom scale built for popup dimensions: 13px body, 16px section labels, minimal heading sizes. Verified at 100% and 125% browser zoom.
Reasoning
Every element readable without scrolling. Hierarchy maintained through size, weight, and colour — not scale alone.

09 — Design System

A token system built for
two themes from day one.

Designing for dark and light simultaneously required a proper semantic token system. Every colour reference uses variables — --bg-primary, --text-secondary — that resolve to different raw values per theme. Theme switching is a variable swap, not a component rebuild.

10 — Design Outcomes

What the design
delivered.

Outcomes are framed against the three problems the design was built to solve — context switching, privacy concerns, and tool complexity. All measured against the documented design assumptions.

⊙
Reduced Context Switching
Full task completion inside one popup — zero tab switches, zero page navigation, zero account creation
◎
Privacy-First Experience
No server request ever made. Privacy is architectural — enforced by design, not relying on a policy document users never read
◐
Faster Workflow
Designed around the 30-second completion goal. Format chips, live preview, and direct download eliminate every unnecessary step
◑
Scalable Foundation
Full token system and component library enable future format additions and feature expansion without visual drift

11 — Future Improvements

Three honest gaps
in V1.2.2.

Gap 01
Validation Testing
Current limitation
All five design assumptions are documented but untested with real users at V1.2.2
Future opportunity
A 3-session guerrilla test — specifically on format chip scanning speed and privacy message trust impact — would validate A1, A2, and A5
Gap 02
Batch Export Optimisation
Current limitation
Batch mode is functionally complete but CSV import error states and progress feedback are underdeveloped
Future opportunity
Inline validation, row-level error reporting, and a progress indicator for large batch jobs would make batch export production-ready for power users
Gap 03
Error State Improvements
Current limitation
Input validation errors use generic messages rather than format-specific guidance
Future opportunity
Format-specific inline error copy — "EAN-13 requires exactly 12 digits before the check digit" — reduces trial-and-error for less-technical users
